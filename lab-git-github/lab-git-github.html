FASE 01 Configurar a Identidade no Git git config ▾
O que é? Antes de qualquer operação, o Git precisa saber quem está a trabalhar. É como assinar um documento: cada commit ficará eternamente com o teu nome e e-mail.
Analogia: Imagina que vais assinar contratos numa empresa. Antes de começar, o departamento regista: "Quem és tu? Qual é o teu e-mail?" — O Git faz exactamente isso.

Abra o terminal (Git Bash no Windows, Terminal no macOS/Linux). Configure o nome do utilizador:

bash
$git config --global user.name "Seu Nome Completo"

Configure o e-mail. Use o mesmo que vai usar no GitHub:

bash
$git config --global user.email "seuemail@exemplo.com"

Configure o editor padrão como VS Code (ou nano, se preferir):

bash
$git config --global core.editor "code --wait"
# Para nano: git config --global core.editor "nano"

Verifique todas as configurações feitas:

bash
$git config --list
user.name=Seu Nome Completo
user.email=seuemail@exemplo.com
core.editor=code --wait

FASE 02 Configurar Chave SSH no GitHub SSH ▾
O que é SSH? SSH (Secure Shell) é um protocolo de comunicação segura. Em vez de digitar a tua password do GitHub cada vez que fazes um push, a chave SSH autentica-te automaticamente — como uma chave física para abrir uma porta.
Analogia: É como ter um passe de segurança. Em vez de mostrar o teu bilhete de identidade e aguardar confirmação, apresentas o passe e a porta abre automaticamente.

Gere um par de chaves SSH (pública + privada):

bash
$ssh-keygen -t ed25519 -C "seuemail@exemplo.com"
Generating public/private ed25519 key pair.
Enter file in which to save the key (~/.ssh/id_ed25519): [Enter]
Enter passphrase (empty for no passphrase): [Enter]

Mostre a chave pública gerada (esta é a que vai para o GitHub):

bash
$cat ~/.ssh/id_ed25519.pub
ssh-ed25519 AAAAC3NzaC1lZDI1NTE5AAAA... seuemail@exemplo.com

Copie toda a linha que aparece.

No GitHub, vá a: Settings → SSH and GPG keys → New SSH key. Cole a chave pública no campo "Key" e guarde.

⚠️ Atenção
A chave privada (id_ed25519) nunca deve ser partilhada. A chave pública (id_ed25519.pub) é a que vai para o GitHub.

Teste a ligação SSH ao GitHub:

bash
$ssh -T git@github.com
Hi username! You've successfully authenticated...

FASE 03 Salvar o Primeiro Projecto no GitHub git init / push ▾
O que fazemos? Criamos uma pasta local, inicializamos o Git nela, criamos um ficheiro, fazemos o primeiro commit e enviamos para o GitHub. Este é o fluxo fundamental: modificar → adicionar → commit → push.
Analogia: É como escrever numa folha de papel (modificar), colocar na pasta certa (stage), carimbar como "versão oficial" (commit) e depois enviar para o arquivo central da empresa (push ao GitHub).

No PC 1: Crie uma pasta e inicialize o Git:

bash
$mkdir projeto-lab
$cd projeto-lab
$git init
Initialized empty Git repository in .../projeto-lab/.git/

Crie um ficheiro index.html simples:

bash
$echo "<!DOCTYPE html><html><body><h1>Olá Git!</h1></body></html>" > index.html

Adicione o ficheiro à área de stage e faça o primeiro commit:

bash
$git add index.html
$git commit -m "primeiro commit: adiciona index.html"
[main (root-commit) a1b2c3d] primeiro commit: adiciona index.html
1 file changed, 1 insertion(+)

No GitHub, crie um repositório vazio chamado projeto-lab (sem README). Copie o URL SSH. Depois ligue e envie:

bash
$git remote add origin git@github.com:SEU_USUARIO/projeto-lab.git
$git branch -M main
$git push -u origin main
Branch 'main' set up to track remote branch 'main' from 'origin'.

FASE 04 Salvar um Novo Commit commit ▾
Repetição deliberada: Nesta secção vamos repetir o ciclo de commit três vezes para solidificar a memória muscular. Cada commit é uma fotografia permanente do estado do projecto naquele momento.

1.º Ciclo — Modifique o index.html, adicione um parágrafo:

bash
$echo "<p>Versão 2 do projecto</p>" >> index.html
$git add index.html
$git commit -m "adiciona parágrafo versão 2"
$git push

2.º Ciclo — Crie um novo ficheiro style.css:

bash
$echo "body { background: #222; color: white; }" > style.css
$git add style.css
$git commit -m "adiciona style.css com tema escuro"
$git push

3.º Ciclo — Adicione todos os ficheiros de uma vez com git add .:

bash
$echo "console.log('olá');" > script.js
$git add .
$git commit -m "adiciona script.js"
$git push

FASE 05 Git Status & Git Log — Inspeccionar o Projecto inspecção ▾
git status mostra o estado actual dos ficheiros (modificados, em stage, não rastreados). git log mostra o histórico de todos os commits feitos. Juntos, são como o painel de controlo do teu projecto.

Modifique o index.html sem fazer commit. Depois execute:

bash
$git status
On branch main
Changes not staged for commit:
modified: index.html

Agora faça git add index.html e execute git status novamente. Observe a diferença!

Veja o histórico completo de commits:

bash
$git log
commit a1b2c3d4e5f6... (HEAD -> main, origin/main)
Author: Seu Nome <seuemail@exemplo.com>
Date: Mon Apr 28 2026
adiciona script.js

Versão compacta e mais legível do log:

bash
$git log --oneline --graph
* a1b2c3d adiciona script.js
* b2c3d4e adiciona style.css
* c3d4e5f primeiro commit
✏️ Anotar no Caderno

  • O que significa HEAD no output do git log?

  • Execute git status antes e depois de um git add e descreva a diferença.

FASE 06 Git Stage, Unstage & Git Diff stage / diff ▾
git diff mostra exactamente o que mudou nos ficheiros. Stage (área de preparação) é onde colocas as modificações antes de comprometê-las com um commit. É possível reverter o stage sem perder o trabalho.
Analogia: O stage é como a bandeja de saída do correio. Colocas as cartas que queres enviar (add). Ainda podes retirá-las (reset) antes de as entregar ao carteiro (commit). O git diff mostra o conteúdo das cartas antes de as fechar.

Modifique o index.html (adicione qualquer texto). Ver o que mudou antes de adicionar ao stage:

bash
$git diff
--- a/index.html
+++ b/index.html
- <h1>Olá Git!</h1>
+ <h1>Olá Git! Actualizado</h1>

Linhas a vermelho (−) foram removidas. Linhas a verde (+) foram adicionadas.

Adicione ao stage e veja o diff staged:

bash
$git add index.html
$git diff --staged
# Mostra as diferenças que ESTÃO em stage (prontas para commit)

Remova do stage sem perder as alterações:

bash
$git restore --staged index.html
# O ficheiro volta a "modified" mas não perde o conteúdo
$git status
modified: index.html

FASE 07 Git Checkout — Navegar no Histórico checkout ▾
git checkout permite-te "viajar no tempo" entre commits ou trocar de branch. É uma das operações mais poderosas do Git — podes ver exactamente como estava o projecto em qualquer ponto da história.
Analogia: É como uma máquina do tempo. Com o hash do commit como "coordenadas", consegues ver o projecto exactamente como estava naquele momento — sem destruir nada do presente.

Primeiro, copie o hash de um commit antigo com o git log:

bash
$git log --oneline
a1b2c3d (HEAD) último commit
b2c3d4e commit anterior
c3d4e5f primeiro commit

Viaje para um commit anterior (substitua pelo hash real do vosso projecto):

bash
$git checkout c3d4e5f
Note: switching to 'c3d4e5f'.
You are in 'detached HEAD' state.
HEAD is now at c3d4e5f primeiro commit

Abra o index.html. O ficheiro está como estava no primeiro commit!

Volte para a versão mais recente (main):

bash
$git checkout main
Switched to branch 'main'

Desfaça modificações não salvas num ficheiro (cuidado: isto apaga o trabalho!):

bash
$git restore index.html
# Ficheiro volta ao estado do último commit. Alterações perdidas!
⚠️ Cuidado
git restore sem --staged apaga as alterações não commitadas permanentemente. Não há undo!
✏️ Anotar no Caderno

  • Qual a diferença entre git checkout [hash] e git checkout main?
  • Qual é a diferença entre git restore --staged e git restore (sem a flag)?

FASE 08 Ficheiro .gitignore .gitignore ▾
.gitignore é um ficheiro especial que diz ao Git quais ficheiros e pastas deve ignorar completamente — como passwords, chaves de API, pastas de dependências (node_modules/) ou ficheiros temporários do sistema operativo.
Analogia: É uma lista negra para o Git. Como um porteiro com lista de não admitidos: mesmo que esses ficheiros existam na pasta, o Git age como se não estivessem lá.

Crie ficheiros que não devem ir para o GitHub:

bash
$echo "DB_PASSWORD=123456" > .env
$mkdir node_modules
$git status
Untracked files: .env, node_modules/

Crie o ficheiro .gitignore:

bash
$cat > .gitignore << EOF
.env
node_modules/
*.log
.DS_Store
EOF

Pode também criar com um editor de texto. Cada linha é um padrão a ignorar.

Verifique que o Git passou a ignorar esses ficheiros:

bash
$git status
Untracked files:
.gitignore
# .env e node_modules já não aparecem!

Commit do .gitignore:

bash
$git add .gitignore
$git commit -m "adiciona .gitignore"
$git push
✏️ Anotar no Caderno
  • Por que nunca devemos commitar um ficheiro .env com passwords?
  • O que faz o padrão *.log no .gitignore?
  • Pesquisem "gitignore.io" — o que é este site? Como pode ajudar?

FASE 09 Clonar e Modificar um Projecto git clone ▾
git clone cria uma cópia completa de um repositório remoto no teu computador — incluindo todo o histórico de commits, branches e ficheiros. É a porta de entrada para contribuir num projecto existente.
🖥️🖥️ Exercício dos Dois Computadores
  • PC 2 vai clonar o repositório que o PC 1 criou
  • PC 2 fará uma modificação e fará push
  • PC 1 fará pull para receber as alterações do PC 2

PC 2: Clone o repositório do PC 1 (peça o URL SSH ao PC 1):

bash — PC 2
$cd ~
$git clone git@github.com:USUARIO_PC1/projeto-lab.git
Cloning into 'projeto-lab'...
done.
$cd projeto-lab
$ls
index.html style.css script.js .gitignore

Para que o PC 2 possa fazer push, o dono (PC 1) deve ir ao GitHub e adicionar o utilizador do PC 2 como colaborador: Settings → Collaborators → Add people.

PC 2: Modifique um ficheiro, commit e push:

bash — PC 2
$echo "<footer>Modificado pelo PC2</footer>" >> index.html
$git add .
$git commit -m "PC2: adiciona footer ao index"
$git push

PC 1: Actualize o repositório local com as alterações do PC 2:

bash — PC 1
$git pull
Updating a1b2c3d..b2c3d4e
Fast-forward
index.html | 1 +
$cat index.html
# O footer do PC2 aparece agora no PC1!

FASE 10 Colaboração — Dois PCs em Paralelo pull / push ▾
Fluxo de trabalho colaborativo: Cada membro deve sempre fazer git pull antes de começar a trabalhar, para garantir que tem a versão mais recente. Isto evita conflitos desnecessários.

Simultaneamente — PC 1 edita o style.css e PC 2 cria um novo ficheiro sobre.html. Cada um faz commit e push:

PC 1 e PC 2 — em paralelo
# PC 1: edita style.css
$git add . && git commit -m "PC1: actualiza estilos" && git push
# PC 2: cria sobre.html
$git pull && git add . && git commit -m "PC2: adiciona sobre.html" && git push

Ambos fazem git pull e confirmam que têm os ficheiros do outro. Verifiquem o log:

bash
$git log --oneline --all
# Devem ver os commits de ambos os PCs!

FASE 11 Deletar Commits reset ▾
Existem duas formas principais de "desfazer" commits:
• git reset --soft — remove o commit mas mantém os ficheiros no stage
• git reset --hard — remove o commit E descarta todas as alterações (irreversível!)
⚠️ Aviso Importante
Usar git reset --hard num repositório partilhado pode causar problemas graves para outros colaboradores. Nunca faças reset de commits que já foram enviados para o GitHub sem avisar a equipa.
Analogia: --soft é como rasgar o carimbo "versão oficial" mas guardar o documento na pasta. --hard é rasgar o carimbo E destruir o documento.

Faça um commit de teste para praticar o reset:

bash
$echo "commit de teste para apagar" > teste.txt
$git add . && git commit -m "commit de teste"
$git log --oneline

Método 1: Apagar o último commit mas manter os ficheiros no stage (--soft):

bash
$git reset --soft HEAD~1
$git status
Changes to be committed: (staged)
new file: teste.txt
# O commit foi apagado, mas teste.txt ainda existe em stage!

HEAD~1 significa "1 commit atrás do actual". HEAD~2 seria 2 commits atrás.

Volte a commitar (para ter algo para testar o hard reset):

bash
$git commit -m "commit de teste (novamente)"

Método 2: Apagar commit E os ficheiros (--hard). Cuidado!

bash
$git reset --hard HEAD~1
HEAD is now at a1b2c3d commit anterior
$ls
# teste.txt desapareceu completamente!

Alternativa segura para desfazer no histórico público — git revert cria um novo commit que desfaz o anterior:

bash
$git revert HEAD
# Abre editor de texto. Salve e feche para confirmar o revert commit.
